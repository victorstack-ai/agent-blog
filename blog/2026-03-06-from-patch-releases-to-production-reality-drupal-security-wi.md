## Agentic Code Requires Execution Proof, Not Model Confidence

Simon Willison's agentic pattern notes and anti-patterns remain the most grounded reference here. Unreviewed agent output shipped to production is how you get mystery bugs with no author. Code generated by an LLM needs to run against real inputs before anyone calls it done.

> "Never assume that code generated by an LLM works until that code has been executed."
>
> — Simon Willison, [Agentic Engineering Patterns](https://simonwillison.net/guides/agentic-engineering-patterns/)

> "Don't file pull requests with code you haven't reviewed yourself."
>
> — Simon Willison, [Anti-patterns: things to avoid](https://simonwillison.net/guides/agentic-engineering-patterns/)

:::warning[Manual testing is now a release gate]
If the workflow ends at "model responded," you have no evidence the code works. Add a mandatory execute-and-verify stage with real commands, fixtures, and failure capture before merge. ~~Prompt quality alone guarantees working code~~ — that idea should have died months ago.
:::

| Pattern | Works in production? | Why |
|---|---|---|
| Generate only | No | No runtime evidence |
| Generate + unit tests only | Sometimes | Misses integration/env drift |
| Generate + execution + review | Yes | Verifies behavior and catches agent hallucinations |

## Drupal 10.6.5 and 11.3.5: Patch Now, Plan Your Migration Window

`10.6.5` and `11.3.5` are production-ready patch releases. Both include CKEditor5 `v47.6.0` updates with security context. The support windows are explicit and non-negotiable: 10.4.x is done, and 10.5.x and 10.6.x have hard deadlines.

> "Drupal 10.4.x security support has ended. Sites on any Drupal version prior to 10.5.x should upgrade to a supported release as soon as possible."
>
> — Drupal.org, [Drupal 10.6.5 release](https://www.drupal.org/project/drupal/releases/10.6.5)

> "Drupal 11.3.x will receive security coverage until December 2026."
>
> — Drupal.org, [Drupal 11.3.5 release](https://www.drupal.org/project/drupal/releases/11.3.5)

| Track | Latest observed patch | Support note | Action |
|---|---|---|---|
| 10.4.x | EOL for security | Ended | Upgrade immediately |
| 10.5.x | Still covered | Until June 2026 | Plan migration to 10.6/11.x |
| 10.6.x | Active | Until Dec 2026 | Stay on current patch |
| 11.3.x | Active | Until Dec 2026 | Patch promptly |

```diff
- "drupal/core-recommended": "^10.4"
+ "drupal/core-recommended": "^10.6.5"
- "drupal/google_analytics": "1.1.13"
+ "drupal/google_analytics": "1.1.14"
- "drupal/calculation_fields": "1.0.3"

```

```bash title="scripts/drupal-security-audit.sh" showLineNumbers
#!/usr/bin/env bash
set -euo pipefail

# highlight-next-line
echo "Checking Drupal core and contrib security posture..."

composer show drupal/core-recommended --format=json | jq -r '.versions[0]'
composer show drupal/google_analytics --format=json | jq -r '.versions[0]'
composer show drupal/calculation_fields --format=json | jq -r '.versions[0]'

# highlight-start
drush pm:security --format=json > build/security-report.json
jq '.[] | select(.advisory != null)' build/security-report.json || true
# highlight-end

echo "Done. Review build/security-report.json before deploy."
```

:::danger[Contrib advisories are not "low priority"]
`SA-CONTRIB-2026-024` (Google Analytics GA4, CVE-2026-3529) and `SA-CONTRIB-2026-023` (Calculation Fields, CVE-2026-3528) are XSS vectors. Patch affected contrib before arguing about severity labels.
:::

Both release pairs follow the same cadence: quick patch follow-ups in active lines, CKEditor security-context updates, and explicit support windows. This is what process maturity looks like in practice — small increments, frequent security hygiene, no last-minute scrambles.

- 10.6.4 and 10.6.5: active maintenance line with support through December 2026
- 11.3.4 and 11.3.5: active 11.x maintenance with same December 2026 coverage note
- Upgrade pressure remains on anything pre-10.5.x

## CISA KEVs, Delta RCE, and Leaked Private Keys: This Week's Concrete Threats

CISA added five KEVs (including Hikvision, Rockwell, and Apple entries), Delta CNCSoft-G2 surfaced an RCE-grade out-of-bounds write, and Google plus GitGuardian published data showing 2,622 valid certs tied to leaked private keys as of September 2025. These are patching and rotation decisions, not things to "monitor."

| Signal | What changed | What to do now |
|---|---|---|
| CISA KEV additions | Active exploitation evidence | Track KEV feed in patch SLA |
| Delta CNCSoft-G2 | Out-of-bounds write, potential RCE | Isolate/segment and patch |
| Key leak telemetry | Valid certs tied to leaked keys | Revoke/rotate automatically |

```yaml title="policy/patch-and-rotate.yaml"
kev_sla:
  critical_exploited: "72h"
  high_exploited: "7d"
certificate_hygiene:
  private_key_leak_detection: true
  auto_revoke_on_match: true
  forced_reissue: true
network_controls:
  ics_segment_isolation: true
  internet_exposed_admin_panels: false
```

Cloudflare's ARR work, QUIC proxy-mode rebuild, and always-on detection efforts are worth studying for the engineering details: removing unnecessary TCP and user-space overhead, keeping detections continuous, and reducing false positives with response-aware signals. Good infra work tends to be boring on purpose.

## What To Do With All of This

Three things to enforce: every AI-assisted change needs runtime verification before merge, every dependency needs a tracked support window, and every security finding needs a dated remediation SLA. That covers most of the ways teams get burned.

:::tip[Single highest-impact move]
Add one pipeline gate this week: block deploy when `drush pm:security` or KEV-mapped dependency checks return unresolved findings. One control, one check, and you eliminate the most expensive category of "we thought it was fine" failures.
:::
